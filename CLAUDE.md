## 코딩 행동 지침

> 속도보다 신중함을 우선합니다. 단순 작업에서는 판단을 사용하세요.

### 1. 코딩 전 먼저 생각하기

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러내세요.**

구현 전:
- 가정은 명시적으로 밝힌다. 불확실하면 질문한다.
- 해석이 여러 가지라면 모두 제시한다 — 조용히 하나를 고르지 않는다.
- 더 단순한 접근법이 있으면 말한다. 필요하다면 반론을 제기한다.
- 불명확한 것이 있으면 멈춘다. 무엇이 혼란스러운지 밝히고 질문한다.

### 2. 단순성 우선

**요청된 문제를 해결하는 최소 코드. 추측성 코드는 금지.**

- 요청 외의 기능 추가 금지.
- 단일 사용 코드에 추상화 금지.
- 요청받지 않은 유연성·설정 가능성 금지.
- 불가능한 시나리오에 대한 에러 핸들링 금지.
- 200줄로 쓴 코드가 50줄로 가능하다면 다시 쓴다.

자문: "시니어 엔지니어가 이 코드를 과복잡하다고 할까?" → 그렇다면 단순화.

### 3. 외과적 변경

**반드시 필요한 것만 건드린다. 자신이 만든 흔적만 정리한다.**

기존 코드 수정 시:
- 인접 코드, 주석, 포맷을 임의로 개선하지 않는다.
- 고장나지 않은 것을 리팩터링하지 않는다.
- 다르게 하고 싶어도 기존 스타일을 맞춘다.
- 관련 없는 데드코드를 발견하면 언급만 한다 — 삭제하지 않는다.

변경으로 고아가 된 것:
- 자신의 변경이 만든 미사용 import/변수/함수는 제거한다.
- 기존 데드코드는 요청 없이 제거하지 않는다.

기준: 변경된 모든 줄이 사용자 요청으로 직접 소급되어야 한다.

### 4. 목표 중심 실행

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환:
- "유효성 검사 추가" → "잘못된 입력 테스트 작성 → 통과시키기"
- "버그 수정" → "재현 테스트 작성 → 통과시키기"
- "리팩터링" → "전후 테스트 모두 통과 확인"

다단계 작업 시 간략한 계획 제시:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```
